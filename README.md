# holder-rank

估算台股股東持股張數在全體股東中的排名。資料來源為台灣集中保管結算所
（TDCC）每週公布的**集保股權分散表**，直接向 TDCC 官方查詢系統
（tdcc.com.tw）取得。適用於任何已在集保登記的上市櫃股票／ETF，不限於
0050，換股票代號即可查詢。

線上版本：https://holder-rank.onrender.com（部署在 Render 免費方案，閒置
一段時間後會休眠，第一次造訪可能要等幾十秒喚醒）。

## API

### `GET /api/rank`

| 參數 | 必填 | 說明 |
|---|---|---|
| `stock` | 是 | 股票代號，例如 `0050`、`2330`（不分大小寫，內部會轉大寫） |
| `lots` | 是 | 持有張數，可為小數（例如 0.5 張 = 500股），必須 > 0 |
| `refresh` | 否 | 設為 `1` 時略過快取，強制重新抓取最新資料 |

**成功回應範例**（`/api/rank?stock=0050&lots=10`）：

```json
{
  "stock": "0050",
  "lots": 10,
  "statDate": "2026-07-03",
  "brackets": [
    { "label": "1-999股", "count": 1409008, "lots": 392552, "pct": 1.89 },
    { "label": "1-5張", "count": 1231519, "lots": 2792117, "pct": 13.45 },
    { "label": "5-10張", "count": 301316, "lots": 2218816, "pct": 10.69 },
    ...
  ],
  "total": { "count": 3273063, "lots": 20754500, "pct": 100 },
  "rank": {
    "N": 3273063,
    "ownBracketLabel": "5-10張",
    "ownBracketCount": 301316,
    "superiorCount": 331220,
    "method": "linear-interpolation",
    "rankEstimate": 331221,
    "percentileEstimate": 10.12,
    "rankRangeLow": 331221,
    "rankRangeHigh": 632536,
    "percentileRangeLow": 10.12,
    "percentileRangeHigh": 19.33
  },
  "cache": {
    "fromCache": true,
    "cachedAt": 1783613423461,
    "ttlMs": 21600000
  }
}
```

`brackets` 依照原始資料順序（人數多的小額級距在前，1,000張以上在最後）；
前端會自行反轉成金字塔順序顯示。`cache` 說明這筆資料是不是從快取來的、快取
時間戳（epoch ms）、以及快取有效期（ms）。

**錯誤回應**（HTTP 400 / 500）：`{ "error": "訊息" }`，常見情況：

- 股票代號查無資料（代號打錯，或該股票沒有集保股權分散表）
- `lots` 缺漏或非正數

### `POST /api/cache/clear`

手動清除快取，body 可帶 `{ "stock": "0050" }` 只清除單一股票，不帶則清空
整個快取。回應 `{ "ok": true, "cleared": "0050" }`。

### `GET /`、`GET /index.html`

回傳首頁，內容經過伺服器端渲染（見下方「首頁 SSR」），不是單純的靜態檔案。

## 資料來源

`fetchDistribution()` 直接對 TDCC 官方查詢系統
（`https://www.tdcc.com.tw/portal/zh/smWeb/qryStock`）發送請求：

1. 先 GET 查詢表單頁面，取出該次請求需要的 session cookie、CSRF用的
   `SYNCHRONIZER_TOKEN`／`SYNCHRONIZER_URI`，以及最新一期的統計日期
   `scaDate`（表單下拉選單的第一個選項）。
2. 用同一組 cookie 與 token，帶著股票代號 POST 查詢，取得該股票的股權
   分散表 HTML。
3. 解析表格：TDCC 用「序 1-15」對應 15 個持股級距（單位是股，不是張），
   逐列轉換成張數（`shares / 1000`），並套用 `BRACKET_ORDER` 裡人類可讀的
   級距名稱（1-999股、1-5張…1,000張以上），最後一列「合計」作為總股東數。
4. 如果查無此股票或表格解析失敗（例如格式不如預期），會丟出對應的錯誤
   訊息（股票代號有誤／解析失敗）。

## 排名估算方法

### 1. 抓資料、解析級距

見上方「資料來源」，`fetchDistribution()` 回傳 `{ statDate, brackets, total }`。

### 2. 定位使用者所在級距

`BRACKET_ORDER` 定義每個級距的 `[lower, upper)` 張數邊界。整數邊界值（5、10、
15…）依照資料本身的分級慣例，歸屬在**較低那個級距的上緣**——例如剛好持有
10 張，算在「5-10張」級距的頂端，而非「10-15張」的底端。

### 3. 級距內插（讓同級距內的不同張數也能分出高低）

如果只用級距邊界算排名，會發生「9張和10張都被丟進同一個5-10張桶子，得到一樣
的名次」這種問題。因此改用內插法，在級距內部依張數位置估算「比你多」的人數：

- **一般級距（有明確上下界）**：假設持股在區間內大致均勻分布，線性內插：

  ```
  比你多的人數(級距內) = 級距人數 × (級距上界 − 你的張數) / (級距上界 − 級距下界)
  ```

  例如「5-10張」共 301,316 人，你持有 9 張：
  `301316 × (10-9)/(10-5) = 60,263`

- **最頂層「1,000張以上」（沒有上界，無法線性內插）**：改用冪律
  （Pareto-style）近似，張數越大估計比例越小：

  ```
  比你多的人數(級距內) ≈ 級距人數 × (1,000 / 你的張數)
  ```

  這是常見的財富分布尾端近似，張數越極端（例如上萬張）誤差可能越大，因為
  沒有實際資料驗證頂層內部的分布形狀。

### 4. 組合成最終名次

```
superiorCount        = 所有比你所在級距「更高」的級距人數總和
rankEstimate          = superiorCount + 級距內插人數 + 1
percentileEstimate    = rankEstimate / N × 100

rankRangeLow/High     = 只用級距邊界（不內插）算出的理論範圍
                        = [superiorCount + 1, superiorCount + 自己級距人數]
```

`rankEstimate` 是主要呈現的估計名次；`rankRangeLow/High` 保留給前端顯示，
讓使用者知道「如果不做內插假設，理論上名次落在哪個範圍」，作為內插估計的
可信度參考。

## 快取機制

TDCC 的股權分散表一週才更新一次，沒必要每次查詢都重新打一次 CSRF token
流程。`server.js` 用一個 `Map<股票代號, { data, fetchedAt }>` 的記憶體快取：

- 預設 TTL 是 6 小時（`CACHE_TTL_MS` 環境變數可調整，單位 ms）。
- 快取有效期內再次查詢同一股票，直接回傳快取資料，不會發出對外請求
  （實測從約 1 秒降到幾毫秒）。
- API 回應會附上 `cache.fromCache`／`cache.cachedAt`，前端會在結果上方顯示
  「資料來自快取（約 X 分鐘前抓取）」或「資料為即時抓取」，並提供一顆
  「重新抓取最新資料」按鈕，會帶 `refresh=1` 強制略過快取。
- 也可以直接呼叫 `POST /api/cache/clear` 手動清快取（見上方 API 說明）。
- 快取只存在記憶體中，重啟伺服器就會清空，不會寫入磁碟或資料庫。
- 首頁 SSR 範例查詢（見下方）跟一般 API 查詢共用同一份快取，所以兩者不會
  互相多打一次對外請求。

## 首頁 SSR（伺服器端渲染範例結果）

單純的 SPA（頁面載入空殼、JS 抓資料後才填內容）有個問題：Google 等搜尋引擎
的爬蟲不會幫你填表單、按按鈕，所以只會看到一個空白頁面，看不到任何真正有
價值的排名資料，SEO 上幾乎沒有內容可以被索引。

為了解決這個問題，`GET /` 與 `GET /index.html` 改成動態路由（註冊在
`express.static` 之前，優先攔截這兩個路徑）：

1. 每次請求時，用預設範例（`DEFAULT_EXAMPLE = { stock: "0050", lots: 10 }`）
   呼叫 `getDistribution()` + `estimateRank()`，跟一般 API 查詢共用同一套
   計算邏輯與快取。
2. 把計算結果透過 `buildResultFragments()` 轉成一組 `{{TOKEN}}` → 實際內容
   的對照表（名次、百分位、金字塔圖表 HTML、級距表格列、快取狀態文字等），
   再用 `renderIndexHtml()` 讀取 `public/index.html` 模板，替換所有 token
   後送出。
3. 結果卡片的標題會加上「範例：」前綴（例如「範例：0050｜持有 10
   張（統計日期：2026-07-03）」），避免使用者誤以為這是自己查的結果。
4. `public/app.js` 的互動邏輯完全不受影響：使用者送出自己的查詢時，前端
   JS 會用 `textContent`／`innerHTML` 整段覆蓋掉 SSR 內容，包含標題的
   「範例：」前綴也會一併被換掉。
5. **容錯**：如果範例查詢當下失敗（TDCC 連線問題等），會自動改用
   `renderIndexHtml(null)` 印出原本的空殼版本（`class="hidden"`），首頁仍
   能正常打開，不會整頁掛掉；錯誤訊息會印在伺服器的 log 裡方便除錯。

`public/index.html` 因此不再是純靜態檔案，而是一份模板：像
`{{RESULT_TITLE}}`、`{{RANK_POINT}}`、`{{PYRAMID_CHART}}`、
`{{BRACKET_ROWS}}` 這些 token 只有經過 `server.js` 的動態路由才會被替換；
如果你看到頁面上顯示原始的 `{{...}}` 字樣，代表繞過了動態路由直接讀了模板
檔案本身。

⚠️ 維護注意：`server.js` 裡的 `renderPyramidChartHtml()` /
`renderBracketTableHtml()` 是刻意仿照 `public/app.js` 裡
`renderPyramidChart()` / `renderResult()` 的渲染邏輯手動重寫一份（沒有共用
模組），**如果之後修改了前端的金字塔圖表或表格渲染方式，記得同步修改
server.js 對應的函式，否則 SSR 版本跟真正互動查詢後的畫面會不一致。**

## 前端呈現邏輯

`public/app.js` 的 `renderResult()`：

1. 把 API 回傳的 `brackets`（小額級距在前）反轉，變成「1,000張以上在最上面、
   1-999股在最下面」的金字塔順序——頂窄（大戶少）、底寬（散戶多）。
2. 由上往下逐層累加人數算出「累計人數」，**只算一次**並附加在每個級距物件上
   （`b.cumulative`），表格欄位與金字塔圖表共用同一組數字，數字會隨著往下層
   越滾越大，最後一列（合計）等於總股東數。
3. 使用者所在的級距列會加上黃底、橘框與「你在這裡」標籤。
4. **持股金字塔圖表**（`renderPyramidChart()`）：把同樣的金字塔順序畫成橫向
   長條圖，長條寬度與顯示的數字都用**累計人數**（而非該級距自己的人數）——
   累計值必然單調遞增，才會呈現乾淨的金字塔形狀，不會因為某個級距人數比上一
   層少而讓圖形凹陷。因為累計人數從幾百（頂端）到上百萬（底端）橫跨好幾個
   數量級，長條寬度改用 **log scale**（而非線性），不然除了最底層以外的長條
   都會細到看不見；使用者所在級距用橘色長條標示。
5. **百分位指示條**（`renderPercentileGauge()`）：左橘右藍的橫向漸層條，
   一個深色圓形游標標出使用者的百分位位置，上方氣泡顯示精確數字。游標的
   **視覺位置**限制在 3%~97% 之間，避免極端百分位（例如 0.0038%）把氣泡
   標籤擠出容器邊緣被裁切；氣泡裡的文字則永遠顯示真實精確的百分位，不受
   視覺限制影響。
6. **常用標的快速選單**：表單下方提供 0050、0056、006208、00878、2330、
   2317、2454、8299 等按鈕，點擊會自動帶入代號並送出查詢，目前選中的標的
   會反白。
7. **鼓勵語與名言小卡**（`renderCheerBox()`）：每次查詢會從固定的語錄庫裡
   隨機挑一句鼓勵語與一句投資名言顯示，純粹是互動查詢時的小彩蛋；首頁 SSR
   版本固定顯示同一組（見上方「首頁 SSR」），避免搜尋引擎每次爬到的內容
   都不一樣。
8. **長期／價值投資觀念提示**（`renderValueTip()`）：跟第7點的隨機小卡不同，
   這個是**依使用者實際的 `percentileEstimate` 分層**、內容固定對應的觀念
   提醒，不是隨機的：

   | 百分位 | 標籤 | 訊息重點 |
   |---|---|---|
   | <1% | 頂層股東 | 別讓部位大小取代判斷力，持續檢視基本面 |
   | 1-10% | 資深股東 | 耐心已見成果，別因短期波動動搖已驗證的判斷 |
   | 10-30% | 穩健累積者 | 紀律比抓時機重要，複利不需要天天盯盤 |
   | 30-70% | 持續累積中 | 複利需要時間發酵，強調持續性 |
   | ≥70% | 剛起步 | 起步不丟臉，價值投資比的是撐過波動的能力 |

   `VALUE_TIP_TIERS` 這份對照表在 `public/app.js`（`getValueTip()`）跟
   `server.js`（`getValueTip()`，給首頁 SSR 用）各自獨立定義了一份、內容
   保持一致——跟前面「首頁 SSR」段落提到的金字塔圖表/表格一樣，**兩邊改
   分層或文案時要記得同步修改**。UI 上用綠色框（跟藍色的名次卡、橘色的
   金字塔圖表做視覺區隔）呈現，位置在百分位指示條正下方，直接呼應使用者
   剛看到的百分位數字。

## 手機 RWD

- `.container` 在 ≤600px 時改成無邊距、無圓角、無陰影的滿版版面（`min-height:
  100vh`），比較像 native app 的呈現方式，而不是桌面版卡片縮小版。
- 表單欄位（股票代號／持有張數）用 `flex: 1 1 140px` 取代固定寬度，
  ≤480px 時改成垂直排列、按鈕滿版寬。
- 表格外層包了 `.table-scroll`（`overflow-x: auto`），小螢幕塞不下 5 欄時
  可以左右滑動查看，表格本身設 `min-width: 460px` 避免欄位被壓爛。
- 金字塔圖表的級距標籤、常用標的按鈕、標題與名次數字，在 ≤480px 時都有
  對應的字體/寬度縮小規則，維持版面平衡。

## SEO 設定

- `<title>`、`meta description`、`canonical` URL。
- Open Graph（`og:title`／`og:description`／`og:type`／`og:url`／
  `og:locale`）與 Twitter Card 標籤，讓分享連結時有正確的標題與描述。
- `google-site-verification` meta 標籤，用於驗證 Google Search Console

- `public/robots.txt`：允許所有爬蟲，並指向 sitemap 位置。

- 最關鍵的一項是上面的「首頁 SSR」：讓爬蟲能在不執行 JS 的情況下，就看到
  真正有內容的頁面，而不只是索引一個空殼表單。

## 注意事項與已知限制

- 排名僅相對於**該檔股票的股東母體**，不是全台灣人口的排名。
- 資料是集保帳戶數，非唯一自然人數，一人多帳戶會使人數略微高估。
- 級距內插假設「持股在區間內大致均勻分布」，這只是合理近似，不是精確排序；
  同一級距內兩個很接近的張數，估計名次也會很接近但不保證完全準確。
- 「1,000張以上」使用冪律近似，沒有實際內部分布資料佐證，數字僅供參考。
- 僅適用於已在集保結算所登記的上市櫃股票／ETF，興櫃或未上市個股沒有股權
  分散表，無法使用此工具。
- 資料每週更新一次（依 TDCC 公告週期），非即時資料；伺服器端另外有 6 小時
  的記憶體快取（見上方「快取機制」），需要最新資料時可點「重新抓取最新資料」
  或帶 `refresh=1`。
- Render 免費方案閒置一段時間會休眠，喚醒需要幾十秒，第一次造訪（包含
  搜尋引擎爬蟲）可能會遇到較慢的回應。

## 套用到其他股票

網頁上直接換「股票代號」欄位即可，後端會用同一套 `BRACKET_ORDER` 與內插公式
計算，不需要修改程式碼。個股（尤其是家族控股集中的公司）跟 0050 這種分散式
ETF 的級距分布形狀會差很多，屬正常現象。
