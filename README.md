# holder-rank

估算台股股東持股張數在全體股東中的排名。資料來源為台灣集中保管結算所
（TDCC）每週公布的**集保股權分散表**，直接向 TDCC 官方查詢系統
（tdcc.com.tw）取得。適用於任何已在集保登記的上市櫃股票／ETF，不限於
0050，換股票代號即可查詢。

主要功能：

- **持股排名估算**：以級距內插法估算你的持股張數在全體股東中的名次與百分位。
- **大戶等級評定**：依估算百分位把你定位到一個評級（S+ / A / B …），附上
  對應的系統評語、鼓勵，以及一句投資大師語錄。
- **分享圖卡**：把大戶等級評定一鍵轉成可下載／分享的 PNG 圖卡，純前端用
  `<canvas>` 產生，不需額外套件或後端支援。
- **即時股價與市值**：向證交所（TWSE）即時報價端點取得現價，換算你目前持股
  的概略市值。
- **持股金字塔圖表**與完整級距表。

線上版本：https://holder-rank.onrender.com（部署在 Render 免費方案，閒置
一段時間後會休眠，第一次造訪可能要等幾十秒喚醒）。

## API

### `GET /api/rank`

| 參數 | 必填 | 說明 |
|---|---|---|
| `stock` | 是 | 股票代號，例如 `0050`、`2330`（不分大小寫，內部會轉大寫） |
| `lots` | 是 | 持有張數，可為小數（例如 0.5 張 = 500股），必須 > 0 |
| `refresh` | 否 | 設為 `1` 時略過快取，強制重新抓取最新資料 |

**成功回應範例**（`/api/rank?stock=0050&lots=10`）：

```json

  "lots": 10,
  "statDate": "2026-07-03",
  "brackets": [

    { "label": "1-5張", "count": 1231519, "lots": 2792117, "pct": 13.45 },
    { "label": "5-10張", "count": 301316, "lots": 2218816, "pct": 10.69 },
    ...
  ],
  "total": { "count": 3273063, "lots": 20754500, "pct": 100 },
  "rank": {
    "N": 3273063,
    "ownBracketLabel": "5-10張",
    "ownBracketCount": 301316,
    "superiorCount": 331220,
    "method": "linear-interpolation",
    "rankEstimate": 331221,
    "percentileEstimate": 10.12,

    "rankRangeHigh": 632536,

    "percentileRangeHigh": 19.33
  },
  "quote": {
    "name": "元大台灣50",
    "market": "tse",
    "price": 105.8,
    "prevClose": 106.05,
    "change": -0.25,
    "changePct": -0.2357,
    "time": "13:30:00"
  },
  "cache": {
    "fromCache": true,
    "cachedAt": 1783613423461,

```

`brackets` 依照原始資料順序（人數多的小額級距在前，1,000張以上在最後）；
前端會自行反轉成金字塔順序顯示。`quote` 是向 TWSE 取得的即時報價（見下方
「即時股價」）；抓取失敗時為 `null`，不影響排名結果。`cache` 說明這筆資料是
不是從快取來的、快取時間戳（epoch ms）、以及快取有效期（ms）。

**錯誤回應**（HTTP 400 / 500）：`{ "error": "訊息" }`，常見情況：

- 股票代號查無資料（代號打錯，或該股票沒有集保股權分散表）
- `lots` 缺漏或非正數

### `POST /api/cache/clear`

手動清除快取（同時清掉股權分散表與即時股價的快取），body 可帶
`{ "stock": "0050" }` 只清除單一股票，不帶則清空整個快取。回應
`{ "ok": true, "cleared": "0050" }`。

此端點會強制觸發一輪對 TDCC／TWSE 的重新抓取，因此**預設停用**：只有在設定了
`CACHE_CLEAR_TOKEN` 環境變數、且請求帶著相符的權杖時才會生效。權杖可放在
`x-cache-clear-token` 標頭或 `?token=` 查詢參數。未設定環境變數時回 403，
權杖不符時回 401。此端點也受下方「速率限制」約束。

### `GET /`、`GET /index.html`

回傳首頁，內容經過伺服器端渲染（見下方「首頁 SSR」），不是單純的靜態檔案。

## 資料來源

`fetchDistribution()` 直接對 TDCC 官方查詢系統
（`https://www.tdcc.com.tw/portal/zh/smWeb/qryStock`）發送請求：

1. 先 GET 查詢表單頁面，取出該次請求需要的 session cookie、CSRF用的
   `SYNCHRONIZER_TOKEN`／`SYNCHRONIZER_URI`，以及最新一期的統計日期
   `scaDate`（表單下拉選單的第一個選項）。
2. 用同一組 cookie 與 token，帶著股票代號 POST 查詢，取得該股票的股權
   分散表 HTML。
3. 解析表格：TDCC 用「序 1-15」對應 15 個持股級距（單位是股，不是張），
   逐列轉換成張數（`shares / 1000`），並套用 `BRACKET_ORDER` 裡人類可讀的
   級距名稱（1-999股、1-5張…1,000張以上），最後一列「合計」作為總股東數。
4. 如果查無此股票或表格解析失敗（例如格式不如預期），會丟出對應的錯誤
   訊息（股票代號有誤／解析失敗）。

### 即時股價（TWSE）

`fetchQuote()` 另外向證交所自家即時報價端點
（`https://mis.twse.com.tw/stock/api/getStockInfo.jsp`）取得現價。同一次請求
會同時帶 `tse_`（上市）與 `otc_`（上櫃）兩個前綴查同一個代號，留下實際有回傳
資料的那一邊，藉此自動判斷該股票屬於上市還是上櫃，不需要事先知道。取得的
現價會用來換算「持股市值 ≈ 張數 × 1000 × 現價」。

即時股價只是排名估算之外的加值資訊，因此包在 `getQuoteSafe()` 裡：抓取失敗
只會讓 `quote` 變成 `null`，不會影響（也不會拖垮）主要的排名結果。

> ⚠️ TDCC 的 `qryStock` 是官方查詢系統；TWSE 的這個報價端點則是「未公開文件、
> 但公開可用」的內部端點，沒有服務保證，理論上有被擋或改版失效的可能。

## 排名估算方法

### 1. 抓資料、解析級距

見上方「資料來源」，`fetchDistribution()` 回傳 `{ statDate, brackets, total }`。

### 2. 定位使用者所在級距

`BRACKET_ORDER` 定義每個級距的 `[lower, upper)` 張數邊界。整數邊界值（5、10、
15…）依照資料本身的分級慣例，歸屬在**較低那個級距的上緣**——例如剛好持有
10 張，算在「5-10張」級距的頂端，而非「10-15張」的底端。

### 3. 級距內插（讓同級距內的不同張數也能分出高低）

如果只用級距邊界算排名，會發生「9張和10張都被丟進同一個5-10張桶子，得到一樣
的名次」這種問題。因此改用內插法，在級距內部依張數位置估算「比你多」的人數：

- **一般級距（有明確上下界）**：假設持股在區間內大致均勻分布，線性內插：

  ```
  比你多的人數(級距內) = 級距人數 × (級距上界 − 你的張數) / (級距上界 − 級距下界)
  ```

  例如「5-10張」共 301,316 人，你持有 9 張：
  `301316 × (10-9)/(10-5) = 60,263`

- **最頂層「1,000張以上」（沒有上界，無法線性內插）**：改用冪律
  （Pareto-style）近似，張數越大估計比例越小：

  ```
  比你多的人數(級距內) ≈ 級距人數 × (1,000 / 你的張數)
  ```

  這是常見的財富分布尾端近似，張數越極端（例如上萬張）誤差可能越大，因為
  沒有實際資料驗證頂層內部的分布形狀。

### 4. 組合成最終名次

```
superiorCount        = 所有比你所在級距「更高」的級距人數總和
rankRangeLow/High     = 只用級距邊界（不內插）算出的理論範圍
                        = [superiorCount + 1, superiorCount + 自己級距人數]
rankEstimate          = clamp(superiorCount + 級距內插人數 + 1,
                              rankRangeLow, rankRangeHigh)
percentileEstimate    = rankEstimate / N × 100
```

`rankEstimate` 是主要呈現的估計名次；`rankRangeLow/High` 保留給前端顯示，
讓使用者知道「如果不做內插假設，理論上名次落在哪個範圍」，作為內插估計的
可信度參考。

`rankEstimate` 會**夾在 `[rankRangeLow, rankRangeHigh]` 之間**：在最頂層
「1,000張以上」的冪律尾端，內插人數會逼近（但不會超過）整個級距人數，四捨
五入後 `+1` 有可能讓點估計比 `rankRangeHigh` 多一名、跑到自己標示的範圍之外，
夾值可避免這種矛盾。這個邊界情況有對應的回歸測試（見「測試」）。

## 快取機制

TDCC 的股權分散表一週才更新一次，沒必要每次查詢都重新打一次 CSRF token
流程。`server.js` 用一個 `Map<股票代號, { data, fetchedAt }>` 的記憶體快取：

- 預設 TTL 是 6 小時（`CACHE_TTL_MS` 環境變數可調整，單位 ms）。
- 快取有效期內再次查詢同一股票，直接回傳快取資料，不會發出對外請求
  （實測從約 1 秒降到幾毫秒）。
- API 回應會附上 `cache.fromCache`／`cache.cachedAt`，前端會在結果上方顯示
  「資料來自快取（約 X 分鐘前抓取）」或「資料為即時抓取」，並提供一顆
  「重新抓取最新資料」按鈕，會帶 `refresh=1` 強制略過快取。
- 也可以直接呼叫 `POST /api/cache/clear` 手動清快取（見上方 API 說明）。
- 快取只存在記憶體中，重啟伺服器就會清空，不會寫入磁碟或資料庫。
- 首頁 SSR 範例查詢（見下方）跟一般 API 查詢共用同一份快取，所以兩者不會
  互相多打一次對外請求。

即時股價另外有一份獨立的短天期快取（`quoteCache`，預設 30 秒 TTL，
`QUOTE_CACHE_TTL_MS` 可調）。股價盤中會變動，但沒必要每次查詢、每次首頁載入
都即時打一次 TWSE——30 秒快取能把大量重複的對外請求（尤其收盤後價格根本不會
動的時段）省下來。抓取失敗的結果也會短暫快取（存成 `null`），避免上游出問題
時被連續重試打爆。

## 速率限制

`/api/rank` 與 `/api/cache/clear` 都套上一層輕量的「每 IP 每分鐘」速率限制
（記憶體計數，預設每分鐘 20 次，`RATE_LIMIT_MAX` 可調），超過回 HTTP 429。

主要目的不是防止惡意攻擊，而是**保護本服務對外那一個 IP**：這些端點每次
查詢都會觸發對 TDCC／TWSE 的請求，其中無效代號還不會進快取（每次都重新
抓）。如果放任有人用大量不同代號猛打，很可能害 Render 的出口 IP 被上游封鎖
——這正是當初要從 norway.twsthr.info 換成 TDCC 的原因。因為部署在 Render 的
反向代理後面，`server.js` 有設定 `trust proxy`，讓 `req.ip` 取得的是真正的
訪客 IP 而不是代理位址。

## 首頁 SSR（伺服器端渲染範例結果）

單純的 SPA（頁面載入空殼、JS 抓資料後才填內容）有個問題：Google 等搜尋引擎
的爬蟲不會幫你填表單、按按鈕，所以只會看到一個空白頁面，看不到任何真正有
價值的排名資料，SEO 上幾乎沒有內容可以被索引。

為了解決這個問題，`GET /` 與 `GET /index.html` 改成動態路由（註冊在
`express.static` 之前，優先攔截這兩個路徑）：

1. 每次請求時，用預設範例（`DEFAULT_EXAMPLE = { stock: "0050", lots: 10 }`）
   呼叫 `getDistribution()` + `estimateRank()`，跟一般 API 查詢共用同一套
   計算邏輯與快取。
2. 把計算結果透過 `buildResultFragments()` 轉成一組 `{{TOKEN}}` → 實際內容
   的對照表（名次、百分位、金字塔圖表 HTML、級距表格列、快取狀態文字等），
   再用 `renderIndexHtml()` 讀取 `public/index.html` 模板，替換所有 token
   後送出。
3. 結果卡片的標題會加上「範例：」前綴（例如「範例：0050｜持有 10
   張（統計日期：2026-07-03）」），避免使用者誤以為這是自己查的結果。
4. `public/app.js` 的互動邏輯完全不受影響：使用者送出自己的查詢時，前端
   JS 會用 `textContent`／`innerHTML` 整段覆蓋掉 SSR 內容，包含標題的
   「範例：」前綴也會一併被換掉。
5. **容錯**：如果範例查詢當下失敗（TDCC 連線問題等），會自動改用
   `renderIndexHtml(null)` 印出原本的空殼版本（`class="hidden"`），首頁仍
   能正常打開，不會整頁掛掉；錯誤訊息會印在伺服器的 log 裡方便除錯。

`public/index.html` 因此不再是純靜態檔案，而是一份模板：像
`{{RESULT_TITLE}}`、`{{RANK_POINT}}`、`{{PYRAMID_CHART}}`、
`{{BRACKET_ROWS}}` 這些 token 只有經過 `server.js` 的動態路由才會被替換；
如果你看到頁面上顯示原始的 `{{...}}` 字樣，代表繞過了動態路由直接讀了模板
檔案本身。

⚠️ 維護注意：因為沒有打包工具、瀏覽器端是純 `<script>` 載入，`server.js`
（SSR）與 `public/app.js`（前端）之間有數處邏輯是**各自手寫一份、靠註解
提醒同步**的，包含：

- `renderPyramidChartHtml()` ↔ `renderPyramidChart()`（金字塔圖表）
- `renderBracketTableHtml()` ↔ `renderResult()` 裡的表格建構
- `formatQuoteLineHtml()`（即時股價那行，兩邊同名）
- `SHAREHOLDER_TIERS` / `getTier()`（大戶等級評定）

**修改其中任一項時，記得兩邊一起改，否則 SSR 首頁跟使用者互動查詢後的畫面
會不一致。** 這是目前已知的技術債（重複程式碼），未來若導入打包工具可以抽成
共用模組消除。

## 前端呈現邏輯

`public/app.js` 的 `renderResult()`：

1. 把 API 回傳的 `brackets`（小額級距在前）反轉，變成「1,000張以上在最上面、
   1-999股在最下面」的金字塔順序——頂窄（大戶少）、底寬（散戶多）。
2. 由上往下逐層累加人數算出「累計人數」，**只算一次**並附加在每個級距物件上
   （`b.cumulative`），表格欄位與金字塔圖表共用同一組數字，數字會隨著往下層
   越滾越大，最後一列（合計）等於總股東數。
3. 使用者所在的級距列會加上黃底、橘框與「你在這裡」標籤。
4. **持股金字塔圖表**（`renderPyramidChart()`）：把同樣的金字塔順序畫成橫向
   長條圖，長條寬度與顯示的數字都用**累計人數**（而非該級距自己的人數）——
   累計值必然單調遞增，才會呈現乾淨的金字塔形狀，不會因為某個級距人數比上一
   層少而讓圖形凹陷。因為累計人數從幾百（頂端）到上百萬（底端）橫跨好幾個
   數量級，長條寬度改用 **log scale**（而非線性），不然除了最底層以外的長條
   都會細到看不見；使用者所在級距用橘色長條標示。
5. **百分位指示條**（`renderPercentileGauge()`）：左橘右藍的橫向漸層條，
   一個深色圓形游標標出使用者的百分位位置，上方氣泡顯示精確數字。游標的
   **視覺位置**限制在 3%~97% 之間，避免極端百分位（例如 0.0038%）把氣泡
   標籤擠出容器邊緣被裁切；氣泡裡的文字則永遠顯示真實精確的百分位，不受
   視覺限制影響。
6. **常用標的快速選單**：表單下方提供 0050、0056、006208、00878、2330、
   2317、2454、8299 等按鈕，點擊會自動帶入代號並送出查詢，目前選中的標的
   會反白。
7. **即時股價與市值**（`formatQuoteLineHtml()`）：結果標題下方顯示一行
   「現價 XX.XX（漲跌／漲跌幅）｜持股市值約 NT$ …（報價時間）」，資料來自
   API 回傳的 `quote`。漲跌用台股慣例的紅漲綠跌配色；沒有報價資料（`quote`
   為 `null`）時這行會自動隱藏。此函式在 `public/app.js` 與 `server.js`
   （SSR）各有一份，需同步維護。
8. **大戶等級評定卡**（`renderTierCard()`）：這是查詢結果唯一的評語／鼓勵
   元素，取代了先前分散的「隨機鼓勵語小卡」「隨機名言」「價值投資提示」三塊。
   依使用者實際的 `percentileEstimate` 把人**分到一個評級**，位置在百分位
   指示條正下方，呼應剛看到的百分位數字，塑造「系統精密計算後給出評分並鼓勵」
   的感覺。每一級包含：評分字母、等級名稱、色調、系統評語、鼓勵，以及一句
   對應的投資大師語錄（附出處）：

   | 百分位（前 X%） | 評分 | 等級名稱 | 色調 |
   |---|---|---|---|
   | <0.5% | S+ | 巔峰巨鯨 | 金 |
   | 0.5-2% | S | 頂級大戶 | 金 |
   | 2-5% | A+ | 資深大戶 | 靛 |
   | 5-12% | A | 資深股東 | 靛 |
   | 12-25% | B+ | 穩健持有者 | 青 |
   | 25-50% | B | 穩定累積者 | 青 |
   | 50-80% | C | 成長新星 | 天藍 |
   | ≥80% | D | 啟程新手 | 石板灰 |

   卡片另外會即時算出「贏過 XX.XX% 的股東」（＝ `100 − percentileEstimate`），
   把「前 X%」正向翻譯成「贏過多少人」，強化精密計算的印象。色調隨評級變化
   （金→靛→青→天藍→灰），讓每個等級有各自的視覺身分。

   `SHAREHOLDER_TIERS` 這份分級對照表在 `public/app.js`（`getTier()`）跟
   `server.js`（`getTier()`，給首頁 SSR 用）各自獨立定義了一份、內容保持
   一致——跟前面「首頁 SSR」段落提到的金字塔圖表／表格一樣，**兩邊改分層
   或文案時要記得同步修改**。

9. **分享圖卡**（`buildShareCanvas()`）：大戶等級評定卡底部的「分享圖卡」
   按鈕，會用 `<canvas>` 即時畫出一張 1080×1350 的直式 PNG（等級、稱號、
   股票／張數、贏過百分比、迷你等級階梯、評語、鼓勵、語錄與網站落款），
   套用跟 `.tier-card--<tone>` 相同的漸層色，畫完後開一個預覽 Modal，提供
   「下載圖片」（`<a download>`）與「分享」（`navigator.share` + 檔案，僅
   支援的瀏覽器/裝置才會顯示這顆按鈕）。

   實作上刻意**直接讀 DOM**（`#rank-percentile` 的文字）重新呼叫
   `getTier()` 算出這張卡要用的資料，而不是去解析畫面上已經格式化好的
   文字，也不依賴前面查詢動作留下的變數——這樣無論是首頁 SSR 的預設範例，
   還是使用者送出查詢後的結果，同一顆按鈕都能正確產生對應內容，不需要在
   `server.js` 額外做鏡射（這是唯一一個**不受**上面「首頁 SSR」段落提到的
   SSR/CSR 同步規則約束的呈現邏輯，因為它本來就是查詢完成、DOM 已經有資料
   之後才會觸發的純前端動作）。

## 測試

`test/estimateRank.test.js` 用 Node 內建的 `node:test`（`npm test` 執行，
不需額外套件）測試核心排名算法。`server.js` 在檔尾用
`if (require.main === module)` 包住 `app.listen`，所以測試可以直接
`require("../server.js")` 取用 `estimateRank`，而不會真的啟動一個伺服器。

測試用一份固定的 0050 股權分散表快照當 fixture（不依賴網路、也不受 TDCC 每週
資料變動影響），涵蓋幾個容易出錯的邊界：

- 剛好持有 1 張要落在「1-5張」而非「1-999股」（回歸測試，這是最早修掉的一個
  單位換算 bug——第一個級距 `1-999股` 的單位是股不是張）。
- 未滿 1 張（例如 0.5 張）要落在「1-999股」。
- 整數邊界（5、10、1000 張）落在較低級距的上緣。
- `rankEstimate` 永遠落在 `[rankRangeLow, rankRangeHigh]` 之內（回歸測試，
  對應上面「排名估算方法」提到的頂層冪律 off-by-one 夾值）。
- 名次、百分位永遠在合理範圍內（1 ≤ rank ≤ N、0 < 百分位 ≤ 100）。
- 持股越多，名次不會反而變差（單調性）。

## 手機 RWD

- `.container` 在 ≤600px 時改成無邊距、無圓角、無陰影的滿版版面（`min-height:
  100vh`），比較像 native app 的呈現方式，而不是桌面版卡片縮小版。
- 表單欄位（股票代號／持有張數）用 `flex: 1 1 140px` 取代固定寬度，
  ≤480px 時改成垂直排列、按鈕滿版寬。
- 表格外層包了 `.table-scroll`（`overflow-x: auto`），小螢幕塞不下 5 欄時
  可以左右滑動查看，表格本身設 `min-width: 460px` 避免欄位被壓爛。
- 金字塔圖表的級距標籤、常用標的按鈕、標題與名次數字，在 ≤480px 時都有
  對應的字體/寬度縮小規則，維持版面平衡。

## SEO 設定

- `<title>`、`meta description`、`canonical` URL。標題／`h1`／
  `meta description`／`og:description`／`twitter:description` 都帶著
  「大戶指數」這個關鍵字——沒有另外加 `<meta name="keywords">`，因為
  Google／Bing 早已不採用這個標籤做排名依據，關鍵字要出現在標題與描述這類
  真正會被爬蟲讀取權重的地方才有效。
- Open Graph（`og:title`／`og:description`／`og:type`／`og:url`／
  `og:locale`）與 Twitter Card 標籤，讓分享連結時有正確的標題與描述。
- `google-site-verification` meta 標籤，用於驗證 Google Search Console
  的網站所有權（因為 `holder-rank.onrender.com` 是 Render 管理的網域，
  沒有 DNS 存取權限，只能用 HTML 標記／檔案這類不需要改 DNS 的驗證方式，
  不能用「網域」資源類型要求的 TXT 記錄）。
- `public/robots.txt`：允許所有爬蟲，並指向 sitemap 位置。
- `public/sitemap.xml`：列出首頁網址，方便主動提交给 Google Search Console。
- 最關鍵的一項是上面的「首頁 SSR」：讓爬蟲能在不執行 JS 的情況下，就看到
  真正有內容的頁面，而不只是索引一個空殼表單。

## 注意事項與已知限制

- 排名僅相對於**該檔股票的股東母體**，不是全台灣人口的排名。
- 資料是集保帳戶數，非唯一自然人數，一人多帳戶會使人數略微高估。
- 級距內插假設「持股在區間內大致均勻分布」，這只是合理近似，不是精確排序；
  同一級距內兩個很接近的張數，估計名次也會很接近但不保證完全準確。
- 「1,000張以上」使用冪律近似，沒有實際內部分布資料佐證，數字僅供參考。
- 僅適用於已在集保結算所登記的上市櫃股票／ETF，興櫃或未上市個股沒有股權
  分散表，無法使用此工具。
- 資料每週更新一次（依 TDCC 公告週期），非即時資料；伺服器端另外有 6 小時
  的記憶體快取（見上方「快取機制」），需要最新資料時可點「重新抓取最新資料」
  或帶 `refresh=1`。
- Render 免費方案閒置一段時間會休眠，喚醒需要幾十秒，第一次造訪（包含
  搜尋引擎爬蟲）可能會遇到較慢的回應。
- 「大戶等級評定」的評語與大師語錄僅為一般性的觀念分享與鼓勵，**不構成任何
  投資建議**，也不針對個人情況；投資決策請自行評估或諮詢合格專業人士。
- 即時股價來自 TWSE 的非公開端點（見「即時股價」），為盤中延遲報價、非官方
  保證資料，僅供參考，不應作為下單依據。

## 套用到其他股票

網頁上直接換「股票代號」欄位即可，後端會用同一套 `BRACKET_ORDER` 與內插公式
計算，不需要修改程式碼。個股（尤其是家族控股集中的公司）跟 0050 這種分散式
ETF 的級距分布形狀會差很多，屬正常現象。
